> 글 성격: Primer — 이 영역의 지형을 보여주는 글. 깊이보다 너비. 독자가 스스로 판단할 수 있는 지도 역할.

## 훅
> "글 1에서 봤듯이 CSR/SSR 중 어느 쪽이 무조건 빠르다는 건 없다. 그러면 렌더링 방식은 어떤 기준으로 골라야 할까?"

## TL;DR 다이어그램 아이디어
```
페이지 특성으로 렌더링 방식 선택

JS 번들  /  서버 연산  /  렌더링 방식 자체
    ↓
각 병목에 맞는 최적화 방향
```

---

## 섹션 구성

### 1. 렌더링 방식은 로딩 속도가 아니라 페이지 특성으로 고른다
- theodorusclarence 4가지 지표
  - 데이터 신선도 (얼마나 최신 데이터가 필요한가)
  - SEO 필요 여부
  - 인증 여부 (로그인 필요 페이지)
  - 업데이트 빈도
- 참고: https://theodorusclarence.com/blog/nextjs-fetch-usecase
- **독자가 얻는 것:** 렌더링 방식 선택의 1차 기준

### 2. 그래도 느리다면 — 병목이 어디냐가 중요하다
- 글 1에서 본 것처럼 일반적 사실이 역전되는 상황이 있다
- 병목 위치에 따라 해결책이 달라진다
- **독자가 얻는 것:** 병목을 먼저 파악해야 한다는 시각

### 3. 병목 유형별 최적화 방향 (간략 소개)
- JS 번들이 크다 → 코드 스플리팅, lazy load
- 서버 연산이 느리다 → 캐싱, edge 활용
- 렌더링 방식 자체가 문제 → 전환 검토 (마지막 수단)
- **독자가 얻는 것:** 더 깊이 알아볼 방향의 지도

## 마무리 방향
> "렌더링 방식은 성능이 아니라 페이지 특성으로 고른다. 느려졌을 때는 병목이 어디인지부터 찾는다. 이 두 가지 원칙을 갖고 있으면, 나머지는 그때그때 찾아서 쓸 수 있다."

## 뺄 내용 (범위 초과)
- 각 최적화 기법 상세 구현 → 기법별 별도 글
- 로딩 UI 구현 가이드 (스피너, 스켈레톤) → 별도 글
- 실제 측정 방법 (Lighthouse, Web Vitals) → 별도 글
- Next.js 구체적 구현 → 별도 글
