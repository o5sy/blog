# 글 1 뼈대: "CSR은 초기 로딩이 느리다 — 이게 항상 맞는 말일까?"

## 훅
> "CSR의 단점으로 초기 로딩이 느리다는 점이 있다. 그런데 SSR도 서버 작업이 많을 경우 초기 로딩이 느려질 수 있다. 상황에 따라 달라지는 내용이라 정리가 필요했다."

---

## TL;DR 다이어그램 아이디어
- CSR vs SSR 타임라인 비교 (TTFB → FCP → TTI)
- 서버 fetch 있는 SSR과 없는 SSR을 나란히

---

## 섹션 구성

### 1. 흔히 알려진 사실
- CSR, SSR의 개념
- CSR/SSR 플로우 차트
- 일반적으로 알려진 장단점 비교 표
- **독자가 얻는 것:** 기준점 잡기

### 2. 의문: SSR도 느릴 수 있지 않나?
- 서버 fetch가 있으면 TTFB가 늦어진다
- 그 사이 CSR은 로딩 UI를 이미 보여준다
- FCP 기준으로는 오히려 CSR이 빠를 수 있는 케이스
- 속도는 상대적 개념 — 무엇을 기준으로, 어느 시점에 측정하느냐에 따라 다르게 해석된다
- **독자가 얻는 것:** "아 나도 이게 이상했었는데"

### 3. "CSR의 초기 로딩이 느리다"는 말의 실체
- 초기 로딩이란 무엇인가 — FCP / TTFB / 체감, 기준마다 의미가 다르다
- 이 말을 성립시키는 암묵적 전제: 서버가 빠르고 JS 번들이 크다
- 전제가 생략된 채 일반화로 굳어진 것
- **독자가 얻는 것:** 일반화된 사실에는 조건이 있다. 그 조건을 알면 더 근거 있는 판단을 할 수 있다

### 4. 초기 로딩이 느리다면 — 병목 지점에 따라 최적화 방향이 달라진다
- TTFB가 느리다 → 서버 연산 최적화 (캐싱, edge 활용)
- JS 번들이 크다 → 코드 스플리팅, lazy load
- 렌더링 방식 전환은 마지막 수단
- 각 항목 상세는 별도 링크/글로 연결
- **독자가 얻는 것:** 느려졌을 때 렌더링 방식을 바꾸기 전에 병목부터 찾는다는 시각

---

## 마무리 방향
> "CSR이 느리다는 건 틀린 말이 아니다. 단지 조건이 붙는 말이다."

조건부 사실임을 정리한 뒤, 두 가지 원칙만 언급하고 마무리:
1. 렌더링 방식은 로딩 속도가 아니라 페이지 특성으로 고른다
2. 느려지면 렌더링 방식을 바꾸기 전에 병목이 어디인지 먼저 찾는다

**더 알아볼 링크 (본문에 자연스럽게 연결)**
- 페이지 특성 기반 선택 기준 → https://theodorusclarence.com/blog/nextjs-fetch-usecase
- 최적화 방향 (코드 스플리팅, 캐싱, edge 등) → 각 기법별 문서 링크 (작성 시 추가)
